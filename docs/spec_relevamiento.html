Página «Verificar Marca» — Herramienta de Relevamiento

Slug objetivo: /verificar-marca  ·  Tipo: página de herramienta (tool page)  ·  Datos SEO: DataForSEO, Argentina (2032), es  ·  Estado: ✅ implementado y testeado (página + proxy + lógica CTA, validado contra el endpoint en vivo)

1Objetivo

Incorporar al dominio principal (unamarca.com.ar) una herramienta para que el usuario final verifique si su marca está disponible para registrar en Argentina. Ante un resultado de búsqueda exitoso, se muestra un botón «Solicitar Registro de Marca» que redirige a WhatsApp con un mensaje pre-cargado.

Por qué dominio principal y no el subdominio: la herramienta ya existe en vigilante.unamarca.com.ar/relevamiento, pero Google trata el subdominio como sitio aparte (casi sin autoridad ni tráfico orgánico). Llevarla al dominio principal consolida la autoridad SEO y conecta la herramienta con el embudo de conversión del blog.

5Evidencia del SERP

SERP de verificar marca registrada, consultar marcas registradas y marca disponible (DataForSEO, AR): el 100% de los resultados son herramientas o bases de datos, no artículos. Google premia formato de herramienta en este carril.

  • Dominan: portaltramites.inpi.gob.ar, argentina.gob.ar, wipo.int, oepm.es.
  • Competidor batible: marcasregistro.com.ar rankea #3 en los tres términos con una tool page titulada «Buscador de Marcas INPI — Consultá Disponibilidad Gratis». De ahí sale el patrón ganador: disponibilidad + gratis + herramienta.
  • Des-canibalización: el post «Cómo saber…» hoy rankea débil (#8) en estas queries de herramienta — un artículo no puede ganarle a una tool aquí. La página dedicada hace que la query correcta la gane la página correcta.
  • Cuidado con marca disponible: dispara AI Overview + People Also Ask (Google absorbe clics). Priorizar verificar / consultar marca registrada, que muestran solo orgánicos.

6Estructura de la página

Página delgada en prosa: el protagonista es la herramienta. La explicación larga vive en los posts del blog (no duplicar aquí, eso recrearía la canibalización).

  1. H1 + sub-línea con la propuesta de valor (gratis, en segundos).
  2. La herramienta (campo de búsqueda + resultado). Above the fold.
  3. Bloque de resultado con el CTA de WhatsApp (ver §7).
  4. Párrafo corto (2–3 líneas) con las frases clave naturales.
  5. FAQ breve (3–4 preguntas) — opcional, apto para rich results.
  6. Enlace a las guías: «¿Querés entender qué significa cada resultado? Leé la guía completa.»

7Contrato de la API + comportamiento de la herramienta

La página NO porta el motor: el engine (search_marcas.py) y la tabla radar_solicitudes viven en el VPS y son la única fuente de verdad. La isla Astro consume el endpoint vía proxy same-origin (ver §11) y renderiza los resultados nativamente.

Endpoint

POST /api/relevamiento/search — origen real https://vigilante.unamarca.com.ar/api/relevamiento/search, consumido como https://unamarca.com.ar/api/relevamiento/search (proxy, §11).

Sin auth, pero requiere header X-Requested-With: XMLHttpRequest o devuelve 403 antes de procesar. El proxy lo inyecta server-side.

Request (JSON)

{
  "marca":         "bilbao",                // REQUERIDO. string, trim, 1–100 chars. (campo = "marca")
  "clases":        ["35","9"],              // REQUERIDO. array de clases como STRINGS "1".."45". Vacío → 400
  "modes":         ["palabras","fonetica","contenida"], // REQUERIDO. subset de los 3
  "min_score":     50,                      // REQUERIDO. int 30–100 (bool/float/fuera de rango → 400)
  "solo_vigentes": false                    // OPCIONAL. bool, default false
}

contenida = substring (tu término dentro de una marca más larga, ej. BILBAO ⊂ BILBAO AMOBLAMIENTOS) y NO está limitada por min_score. No hay modo «todas las clases»: la búsqueda es por clase; para cubrir todo hay que enviar las 45 (costo lineal por clase — ver §3, limitar del lado cliente).

Response 200 (JSON) — exactamente 9 campos por resultado

{
  "results": [{
    "acta":               "3456789",              // nro INPI (string|number|null) → /inpi/acta/<acta>
    "clase":              "35",                   // string — agrupar las tarjetas por esto
    "denominacion":       "BILBAO AMOBLAMIENTOS", // nombre de la marca en conflicto
    "estado_label":       "Concedida",            // label humano (enum abajo)
    "fecha_presentacion": "2018-06-12",           // ISO o null
    "fecha_vencimiento":  "2028-06-12",           // ISO o null
    "vigente":            true,                   // bool: vencimiento >= hoy
    "score":              92.0,                   // float 0–100 (bajo a propósito en "contenida")
    "match_type":         "fuzzy"                 // exact|fuzzy|token_sort|containment|phonetic|contenida
  }]
}

titular, tipo y nro_resolucion existen internamente pero se eliminan de la respuesta — NO disponibles para el markup. Server ya deduplica por acta (mejor score), ordena vigentes-primero y luego score desc; cap de 100 por clase.

Configuración del formulario (decidido).
  • Clases: el usuario elige clase(s) específicas o «Todas» (reusar el selector de la v2 con su checkbox «Todas»). «Todas» = enviar las 45.
  • Modes: ocultos. Se envía siempre ["palabras","fonetica"]. Sin contenida.
  • Slider de % similitud: visible (default 50%). Controla qué resultados se muestran.
  • Sólo vigentes: a criterio — puede ocultarse con default false.

Lógica de conversión — doble evaluación (display vs CTA)

Dos umbrales distintos sobre la misma búsqueda:

  • Display: se muestran los resultados con score ≥ valor del slider (lo que el usuario ve, agrupado por clase).
  • CTA: el texto y la acción del botón dependen SIEMPRE de la evaluación a umbral 60 («búsqueda oculta»), independiente del slider.
Optimización — una sola llamada (importa por el rate-limit). No hace falta pegarle dos veces al API. Pedir min_score = min(slider, 60), mostrar results.filter(r => r.score ≥ slider) y evaluar el CTA sobre los results con score ≥ 60. Conducta idéntica a «dos búsquedas», mitad de llamadas.

Regla del botón

// Set de evaluación del CTA: score>=60, VIGENTE, y EXCLUYENDO phonetic
const ctaSet = results.filter(r =>
  r.score >= 60 && r.vigente && r.match_type !== "phonetic");
const maxScore = Math.max(0, ...ctaSet.map(r => r.score));

if (maxScore >= 80) {
  // parecido fuerte, vigente y NO fonético → rutear a humano
  boton = "Contactarnos por WhatsApp";
} else {
  // nada fuerte (incluye 0 resultados) → empujar conversión directa
  boton = "Solicitar Registro Inmediatamente";
}

Borde: score == 80 cuenta como conflicto (>= 80 → Contactarnos).

Evaluación del CTA (decidido): solo cuentan resultados vigentes y se excluye phonetic (puntúa fijo 85, inflaba el conflicto). Así «Solicitar Registro Inmediatamente» aparece salvo que exista un parecido fuerte real (fuzzy/token_sort/exact ≥ 80) sobre una marca con registro vigente. Importante: el display (slider) sigue mostrando los fonéticos — esta exclusión afecta solo qué botón se muestra, no qué resultados ve el usuario.

Links a INPI (acta)

/inpi/acta/<acta> NO es un 302: INPI solo acepta POST, así que la ruta devuelve un mini-HTML con un form auto-submit. Linkear cada resultado a https://vigilante.unamarca.com.ar/inpi/acta/<acta> con target="_blank" y dejar que el bounce haga el POST. No se puede deep-linkear INPI con GET. (Alternativa: replicar el mismo form auto-POST en nuestro propio origen — es autocontenido, sin secretos.)

Pre-cargar el término buscado hace que el lead llegue calificado y tibio. El click en el botón es el evento de conversión (ver §9).

11Arquitectura de integración (API remota + proxy)

El motor y la data se quedan en el VPS. La página estática los consume vía HTTP. Decisión central: cómo cruzar de origen, ya verificado con DNS en vivo.

Topología confirmada (dig + headers, junio 2026)

  • unamarca.com.ar → nameservers Cloudflare, IPs Cloudflare, server: cloudflare + cf-ray. Está detrás de Cloudflare (Pages).
  • vigilante.unamarca.com.ar → 46.224.159.63 (Hetzner), server: nginx, sin cf-ray. DNS-only, directo al VPS. X-Frame-Options: DENY → iframe bloqueado (la API es el único camino).

El problema del cruce de origen

Fetch directo unamarca.com.ar → vigilante.unamarca.com.ar falla: no hay headers CORS, y X-Requested-With es un header no-simple que dispara un preflight OPTIONS que la ruta no maneja → falla. No es opcional resolverlo.

Solución recomendada — proxy same-origin vía Cloudflare Pages Function

El proxy vive del lado de unamarca (Cloudflare Pages), NO en el nginx de vigilante. Una Pages Function reescribe unamarca.com.ar/api/relevamiento/* → VPS, inyecta el header server-side, y el browser hace una llamada same-origin (sin CORS, sin preflight).

// functions/api/relevamiento/search.js (Cloudflare Pages Function)
export async function onRequestPost({ request }) {
  const body = await request.text();
  const upstream = await fetch(
    "https://vigilante.unamarca.com.ar/api/relevamiento/search",
    { method: "POST",
      headers: {
        "Content-Type": "application/json",
        "X-Requested-With": "XMLHttpRequest"  // inyectado acá
      },
      body });
  return new Response(upstream.body, {
    status: upstream.status,
    headers: { "Content-Type": "application/json" }
  });
}

Mismo patrón para /inpi/acta/<acta> si se decide no linkear directo a vigilante.

Rate limiting — moverlo a Cloudflare

El flask-limiter del VPS (20/hora por IP) no sirve detrás del proxy: sin ProxyFix, get_remote_address ve una sola IP de egress de Cloudflare → límite global, no por usuario. Además es memory:// por worker (×2 con -w 2, se borra al reiniciar). Solución: una regla de Rate Limiting de Cloudflare sobre /api/relevamiento/*. Eso protege la tabla SQLite y evita tocar el plumbing de IP real en el VPS. El peor caso a limitar: muchas clases + modo contenida.

Alternativa (si no se usa Cloudflare Pages Function)

Agregar flask-cors (no instalado hoy) scoped a /api/relevamiento/* para el origen de unamarca, manejar el preflight OPTIONS y echo de X-Requested-With en Access-Control-Allow-Headers. Más piezas móviles — preferir el proxy.
